# 坐标模式下线与符号模式增强实施文档

## 1. 背景与动机

### 1.1 当前两种定位模式

CLI 的 6 个 LSP 命令（`def`/`refs`/`hover`/`impl`/`typedef`/`ch`）同时支持两种定位模式：

| 模式 | 用法 | 原理 |
|------|------|------|
| **坐标模式** | `jls def <file> <line> <col>` | 用户手动指定行列号，直接传给 JDT LS |
| **符号模式** | `jls def <file> --symbol <name>` | 用户指定符号名 → 内部解析坐标 → 传给 JDT LS |

### 1.2 坐标模式存在的问题

2026-04-28 实测发现：**坐标模式不可靠**。

```
# 坐标模式 - 返回空结果
$ jls def "SqlSession.java" 32 40
{ "success": true, "data": [] }   ← JDT LS 未找到定义

# 符号模式 - 正常工作
$ jls def "SqlSession.java" --symbol "selectOne" --signature "(String)"
{ "success": true, "data": [{ "uri": "file:///...", "range": {...} }] }
```

**根本原因**：JDT LS 对行列号精度要求极高，用户手动数行列极易出错（1-based/0-based 混淆、制表符宽度差异等），而符号模式通过 LSP `documentSymbol` 精确确定位置。

### 1.3 目标

1. **下线坐标模式**：从 CLI 接口移除 `[line] [col]` 位置参数，统一使用符号模式
2. **增强输出**：在响应 `metadata` 中返回解析后的 `resolvedPosition`，让调用方能拿到精确坐标

## 2. 代码逻辑关系全景

### 2.1 调用链路

```
cli.ts (入口)
  └─ program.command('def [file] [line] [col]')  ← 注册命令
       └─ .action(async (file, line, col, cmdOptions) => {
            └─ getPosition(file, line, col, cmdOptions, opts)  ← 位置解析枢纽
                 ├─ 坐标模式：直接校验 line/col → 返回 { filePath, line, col }
                 └─ 符号模式：cmdOptions 有 --symbol？
                      └─ resolvePositionBySymbol() → 调 documentSymbol → 匹配 → 返回坐标
            └─ executeCommand(endpoint, body, directHandler, opts, commandName)
                 ├─ 守护进程模式：sendDaemonRequest()
                 └─ 直接模式：directHandler()
                      └─ outputResult()  ← 最终输出
```

### 2.2 关键模块职责

```
src/cli/commands/          ← 6 个命令定义文件（各含坐标参数注册）
  ├── definition.ts        ← def [file] [line] [col]
  ├── references.ts        ← refs [file] [line] [col]
  ├── hover.ts             ← hover [file] [line] [col]
  ├── implementations.ts   ← impl [file] [line] [col]
  ├── typeDefinition.ts    ← typedef [file] [line] [col]
  └── (call-hierarchy 在 commandHandlers.ts)

src/cli/commandHandlers.ts ← ch [file] [line] [col] (+ 增强模式)
src/cli/utils/positionResolver.ts ← getPosition() 位置解析核心
src/cli/utils/outputHandler.ts    ← outputResult() 输出格式化
src/core/types.ts                 ← ResponseMetadata 元数据类型
```

### 2.3 不受影响的命令

| 命令 | 原因 |
|------|------|
| `symbols` / `sym` | 只需要 `<file>`，不涉及坐标定位 |
| `find` / `f` | 全局搜索，只接受 `<query>` 字符串 |
| `config` | 配置管理，不涉及 LSP 请求 |
| `daemon` | 守护进程管理，不涉及 LSP 请求 |

### 3.2 核心逻辑变更：`getPosition()`

> **策略**：坐标模式分支代码以注释方式下线（保留完整代码），而非物理删除。如需恢复坐标模式，只需取消注释并调整入口签名即可。

**变更前** (`positionResolver.ts` L282-L431)：

```typescript
export async function getPosition(
  file, line, col, cmdOptions, opts   // line, col 是可选参数
): Promise<{ filePath, line, col } | CLIResult> {
  // 1. cursor 续查模式 → 跳过
  // 2. --global 模式 → 全局搜索
  // 3. 符号模式 → resolvePositionBySymbol()
  // 4. 坐标模式 → 校验 line/col → 直接返回   ← 要注释下线的内容
}
```

**变更后**：

```typescript
export async function getPosition(
  file, cmdOptions, opts              // line, col 参数从入口移除
): Promise<{ filePath, line, col } | CLIResult> {
  // 1. cursor 续查模式 → 跳过
  // 2. --global 模式 → 全局搜索
  // 3. 符号模式（唯一路径）→ resolvePositionBySymbol()
  //    如果未提供 --symbol/--method → 返回明确错误提示

  /*
   * [已下线] 坐标模式分支（原 L377-L430）
   * 原因：用户手动数行列极易出错（1-based/0-based 混淆、制表符宽度差异等），
   *       JDT LS 对坐标精度要求极高，实测返回空结果概率高。
   * 保留策略：代码完整保留但注释掉，如需恢复取消注释即可。
   *
   * if (line !== undefined && col !== undefined) {
   *   // ... 坐标模式校验逻辑 ...
   *   return { filePath, line, col };
   * }
   */
}
```

### 3.3 类型定义变更：`ResponseMetadata`

**变更前** (`core/types.ts` L364-L384)：

```typescript
export interface ResponseMetadata {
  compactMode?: boolean;
  childrenExcluded?: boolean;
  totalSymbols?: number;
  depthLimit?: number;
  projectStatus?: ProjectStatus;
  diagnosis?: { ... };
}
```

**变更后**：

```typescript
export interface ResponseMetadata {
  compactMode?: boolean;
  childrenExcluded?: boolean;
  totalSymbols?: number;
  depthLimit?: number;
  projectStatus?: ProjectStatus;
  diagnosis?: { ... };
  /** 符号模式解析后的精确位置 */

    col: number;     // 1-based 列号
  };
}
```

### 3.4 输出格式变更

**变更前**（符号模式输出）：
```json

  "data": { "references": [...], "count": 10 },
```

## 4. 逐文件修改详情

### 4.1 `src/cli/utils/positionResolver.ts`

**注释下线**（非物理删除）：L377-L430 整个坐标模式分支。
- 用 `/* ... */` 块注释包裹，并添加下线说明注释（原因、时间、恢复方式）
- 代码完整保留在源文件中，便于后续复用或恢复

**修改**：函数签名从 `getPosition(file, line, col, cmdOptions, opts)` 改为 `getPosition(file, cmdOptions, opts)`。（`line`/`col` 参数从入口移除，下游 `resolvePositionBySymbol()` 内部仍可使用同名变量。）

**修改**：符号模式未提供 `--symbol`/`--method` 时，错误信息更新：
```
旧：'Position required: either provide <line> <col> or use --method/--symbol option'
新：'Position required: use --symbol <name> [--signature <sig>] [--kind <kind>]'
```

**保留不变**（下游链路全部保持）：
- `--cursor` 续查模式跳过逻辑（L291-L298）
- `--global` 全局搜索逻辑（L302-L334）
- `resolvePositionBySymbol()` 符号解析逻辑（L369-L374）
- `executeCommand()` 和 `createDirectClient()`
- 坐标模式被注释的代码完整保留在文件中

### 4.4 `src/cli/utils/outputHandler.ts`

**修改位置** `outputResult()` 函数（L87-L125）：

在构建输出 JSON 之前，注入 `resolvedPosition`。需要在调用链中传递解析后的位置信息。

**方案**：`executeCommand()` 在调用 `outputResult()` 之前，从 `body` 参数中提取 `file`/`line`/`col`，注入到结果的 `metadata.resolvedPosition` 中。

**具体改动**：在 `positionResolver.ts` 的 `executeCommand()` 函数中（L452-L511），在调用 `outputResult()` 前注入 `resolvedPosition`：

```typescript
// 在 outputResult 调用前
const resolvedFile = body.file;
const resolvedLine = body.line;
const resolvedCol = body.col;

// 注入到结果中
const enrichedResult = {
  ...result,
  metadata: {
    ...(result.metadata || {}),
    resolvedPosition: resolvedFile !== '__cursor_mode__' ? {
      file: resolvedFile,
      line: typeof resolvedLine === 'string' ? parseInt(resolvedLine) : resolvedLine,
      col: typeof resolvedCol === 'string' ? parseInt(resolvedCol) : resolvedCol,
    } : undefined,
  },
};
```

## 8. 风险与回滚

| 风险 | 缓解措施 |
|------|---------|
| 用户脚本依赖坐标模式 | 错误提示中引导使用 `--symbol`，提供迁移示例 |
| `--symbol` 歧义（多匹配无 `--index`） | 已有完善的歧义提示机制，列出候选 |
| `--global` 搜索性能 | 已有 `--kind` 强制要求，减少不必要搜索 |

### 8.1 回滚方式

由于坐标模式代码以注释方式保留（非物理删除），回滚非常简单：

1. **恢复坐标模式**：在 `positionResolver.ts` 中取消坐标模式分支的注释
2. **恢复入口签名**：6 个命令文件 + `commandHandlers.ts` 将 `.command('xxx <file>')` 改回 `.command('xxx [file] [line] [col]')`，action 函数恢复 `line`/`col` 形参

全部改动集中在 8 个源文件中，也可直接 `git revert`。
