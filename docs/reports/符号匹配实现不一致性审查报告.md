# 符号匹配实现不一致性审查报告

审查日期: 2026-04-27
触发来源: 测试审查报告.md 复核时发现
严重程度: 🔴 高（影响线上行为的代码无测试覆盖）

## 一、问题概述

项目中存在 **三套并行的符号匹配实现**，行为互不相同，但只有其中一套有单元测试覆盖。
真正在 Daemon 和 CLI 运行时使用的代码路径反而**零测试覆盖**，已测试的版本并未被线上逻辑调用。

### 三套实现的位置

| # | 文件路径 | 用途 | 测试覆盖 |
|---|---------|------|---------|
| A | `src/services/symbolService.ts` | 独立的服务类，导出 `SymbolService` 等 | ✅ 有完整测试 |
| B | `src/symbolResolver/` 目录下各模块 | 被 Daemon/CLI 实际调用的版本（通过 `src/symbolResolver.ts` 重导出） | ❌ 零测试 |
| C | `src/cli/utils/positionResolver.ts` 中的 `matchSignature` | CLI 侧 `resolveGlobalPosition` 内部使用 | ❌ 无测试 |

### 实际调用链

```
Daemon 请求 → routeHandlers.ts → daemon/positionResolver.ts
                                     ↓
                              import { resolveSymbol, buildSymbolQuery } from '../../symbolResolver'
                                     ↓
                              symbolResolver.ts → symbolResolver/core/symbolResolver.ts
                                                     ↓
                                              symbolResolver/matching/signatureMatcher.ts ← 无测试！
                                              symbolResolver/signature/signatureExtractor.ts ← 无测试！

CLI 命令 → cli/positionResolver.ts
              ↓
           import { resolveSymbol, buildSymbolQuery, isSymbolMode } from '../../symbolResolver'  ← 同上
           自带 matchSignature()  ← 第三套实现，无测试！
```

**单元测试覆盖的路径**:
```
test/unit/services/symbolService.test.ts
    ↓
import { matchSignature, fuzzyMatchName, SymbolService } from '../../../src/services/symbolService'  ← 版本A
```

## 二、具体行为差异

### 2.1 `fuzzyMatchName` — 版本A vs 版本B

| 特性 | 版本A: `symbolService.ts` (有测试) | 版本B: `symbolResolver/matching/signatureMatcher.ts` (无测试) |
|------|------|------|
| 两个空字符串 `fuzzyMatchName('', '')` | `true` | `false` |
| 一方为空 `fuzzyMatchName('method', '')` | `false` | `false` |
| 大小写不敏感 `fuzzyMatchName('mymethod', 'MyMethod')` | ✅ `true` | ❌ `false` |
| 子串匹配 `fuzzyMatchName('Method', 'myMethod')` | ✅ `true` | ❌ `false` |
| 驼峰↔下划线 `fuzzyMatchName('process_order', 'processOrder')` | ✅ `true` | ❌ `false` |
| 前缀匹配 | ✅ 双向 | ✅ 双向 |

**影响**: Daemon/CLI 中 `fuzzyMatchName('mymethod', 'MyMethod')` 返回 `false`，可能导致符号查找失败。
而单元测试中验证了大小写不敏感功能，但那测的是不被调用的版本A。

### 2.2 `matchSignature` — 三个版本对比

| 签名匹配策略 | 版本A: `symbolService.ts` | 版本B: `signatureMatcher.ts` | 版本C: `positionResolver.ts` |
|-------------|------|------|------|
| 精确匹配（保留泛型） | ✅ | ✅ | ❌ |
| 模糊匹配（移除泛型） | ✅ | ✅ | ❌ |
| 部分匹配 | `startsWith` (前缀) | `includes` (双向子串) | `includes((...))` (括号内包含) |
| 从 symbolName 提取签名 | ✅ 支持 | ✅ 支持 | ❌ 不支持 |
| symbolDetail 本身是签名 | ✅ 回退处理 | ❌ 无此逻辑 | ❌ 无此逻辑 |

**关键差异**: `matchSignature('String, int', 'int')`

- 版本A → `false`（`'string,int'.startsWith('int')` = false）
- 版本B → `true`（`'string,int'.includes('int')` = true）
- 版本C → 不适用（需要括号格式）

单元测试中断言 `matchSignature('String, int', 'int')` 应返回 `false`，但此断言只验证了版本A。
线上运行的版本B对此输入会返回 `true`，行为不一致。

### 2.3 `normalizeGenericType` — 嵌套泛型处理

```ts
// 版本A: symbolService.ts — 循环处理，正确
let result = typeName;
let prevResult;
do {
  prevResult = result;
  result = result.replace(/<[^<>]*>/g, '');
} while (result !== prevResult);
return result.replace(/<.*$/g, '').trim();

// 版本B: symbolResolver/signature/signatureExtractor.ts — 单次替换，深层嵌套失败
return typeName.replace(/<[^<>]*>/g, '').replace(/<.*$/g, '').trim();
```

**影响示例**: `Map<String, List<Map<String, Integer>>>`

- 版本A → `Map` ✅（循环剥洋葱式移除）
- 版本B → 可能残留不完整结果（单次替换只处理最内层）

### 2.4 `normalizeSignature` — 泛型内逗号处理

```ts
// 版本A: symbolService.ts — smartSplitSignature 正确处理泛型内逗号
function smartSplitSignature(signature: string): string[] {
  // 跟踪 <> 嵌套深度，只在 depth=0 时按逗号分割
  // "Map<String, Integer>, List<String>" → ["Map<String, Integer>", "List<String>"]
}

// 版本B: symbolResolver/signature/signatureExtractor.ts — 简单 split
signature.split(',')  // "Map<String, Integer>, List<String>" → ["Map<String", " Integer>", " List<String>"] ❌
```

**影响**: 当签名包含泛型参数时（如 `Map<String, Integer>`），版本B会错误地在泛型内逗号处分割，
导致签名匹配逻辑出错。

## 三、根因分析

### 为什么存在两套实现

1. **`symbolService.ts`** 是早期单体文件，包含了所有符号解析逻辑
2. 后来重构时将功能拆分到 **`symbolResolver/`** 目录下的模块化架构
3. 重构时 `symbolService.ts` 没有被删除，保留了原有代码作为"独立服务类"
4. 两个版本独立演化，行为逐渐分歧
5. 单元测试始终针对 `symbolService.ts` 编写，未跟进重构后的调用链变化

### 为什么版本B功能更弱

重构拆分时，`signatureMatcher.ts` 和 `signatureExtractor.ts` 只实现了核心逻辑，
未将 `symbolService.ts` 中后续增强的功能（大小写不敏感、子串匹配、驼峰转换、智能分割等）同步过来。

---

## 四、修复方案

### 方案一：统一到 `symbolResolver/` 模块（推荐）

1. 将 `symbolService.ts` 中的增强功能迁移到 `symbolResolver/` 对应模块：
   - `fuzzyMatchName`: 补充大小写不敏感、子串匹配、驼峰↔下划线
   - `normalizeGenericType`: 改为循环处理嵌套泛型
   - `normalizeSignature`: 改为 `smartSplitSignature` 处理泛型内逗号
   - `matchSignature`: 统一部分匹配策略（建议用 `startsWith`，更可控）
2. 删除 `symbolService.ts` 中的重复实现，改为从 `symbolResolver/` 重导出
3. 删除 `positionResolver.ts` 中的第三套 `matchSignature`，改为导入 `symbolResolver/` 版本
4. 将现有测试的 import 路径改为指向 `symbolResolver/` 模块
5. 为 `symbolResolver/` 各模块补充独立的单元测试

### 方案二：保留两套但增加集成测试

不合并代码，但确保两套实现的关键行为一致，增加集成测试覆盖 Daemon/CLI 实际调用路径。

## 五、影响范围评估

| 场景 | 受影响功能 | 严重程度 |
|------|-----------|---------|
| 用户输入小写方法名 `--method selectone` | `fuzzyMatchName` 大小写不匹配 → 符号查找失败 | 🟡 中 |
| 签名含泛型 `--signature "Map<String, Integer>"` | `normalizeSignature` 错误分割 → 签名匹配失败 | 🔴 高 |
| 深层嵌套泛型 `List<Map<String, Integer>>` | `normalizeGenericType` 处理不完整 → 匹配失败 | 🟡 中 |
| 部分签名查询 `--signature "int"` | `matchSignature` 策略不同 → 可能误匹配 | 🟡 中 |
| CLI 全局定位 `--global --symbol` | `positionResolver.ts` 的 `matchSignature` 行为完全不同 | 🟡 中 |
